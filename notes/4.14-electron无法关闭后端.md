### 根因分析

看了 [`frontend/electron/managers/backendManager.js`](frontend/electron/managers/backendManager.js) 的代码，残留进程主要来自 **3 个缺陷**：

#### 1. Windows 下 `taskkill` 没有杀子进程树
[`killProcess`](frontend/electron/managers/backendManager.js:100-115) 在 Windows 上执行的是：
```js
exec(`taskkill /F /PID ${pid}`, ...)
```
这只会杀掉指定的 PID，**不会递归杀掉它的子进程**。  
你的后端是 `python main.py`，而 `main.py` 内部大概率用 `uvicorn` 启动，uvicorn 会再 fork 出工作进程。这些子进程在父进程被 `/F` 强制杀死后变成了**孤儿进程**，继续霸占 `8000` 端口。

#### 2. `stop()` 是 callback 风格，调用方没有等待
[`stop(callback)`](frontend/electron/managers/backendManager.js:65) 内部是异步的，但 [`main.js`](frontend/electron/main.js:28-38) 里这么写：
```js
app.on('window-all-closed', () => {
  backendManager.stop();
  terminalManager.killAllTerminals();
  app.quit();
});
```
`backendManager.stop()` 刚发完 kill 命令，还没等进程真正退出，Electron 就执行了 `app.quit()`，导致后续的 `checkAndKill` 和端口清理逻辑被中断。

#### 3. `restart()` 存在竞态（race condition）
```js
restart() {
  this.stop(() => {
    this.start();
  });
}
```
`stop` 的 callback 是在发现端口上没有 PID 时才触发的，但如果 kill 命令执行后、端口释放前的时间窗口内，新的 `start()` 已经被调用，就会出现**旧进程还没死透，新进程已经开始监听**，最终端口上挂了 N 个进程。

---

### 解决思路

| 问题 | 修复方案 |
|------|---------|
| **子进程残留** | Windows 下 `taskkill` 加上 `/T` 参数（`taskkill /F /T /PID ${pid}`），强制递归杀整棵进程树。 |
| **调用方没等待** | 把 `stop()` 从 callback 风格重构为 **返回 `Promise`**，让 `main.js` 在退出事件里 `await backendManager.stop()`。 |
| **重启竞态** | `restart()` 内部 `await this.stop()`，然后增加一个**轮询等待端口释放**的逻辑（最多等几秒），确认端口彻底没人用了再 `start()`。 |
| **孤儿进程兜底** | 保留端口扫描作为兜底，但扫描到 PID 后也统一用带 `/T` 的 `taskkill` 杀掉。 |

### 额外建议
如果你希望更彻底地避免残留，还可以在 `spawn` 时设置：
```js
windowsHide: true,
env: { ...process.env, UVICORN_RELOAD: 'false' }
```
并检查后端 `uvicorn.run()` 是否开启了 `reload=True`（reload 模式会额外产生文件监控子进程，也会加大残留概率）。

## PID 为 0 的「东西」是什么？

**PID 为 0 的连接是 `TIME_WAIT` 状态的 TCP 幽灵连接，由 Windows 内核直接托管，不隶属于任何用户进程。**

当 TCP 连接完成四次挥手后，主动关闭方会进入 `TIME_WAIT` 状态并持续一段时间（Windows 默认 60~120 秒），用于确保最后一条 ACK 不会丢失。因为它没有所属进程，PID 显示为 0，**你杀不掉它，也无需去杀**——系统会在 1~2 分钟内自动回收。

## 为什么反复重启会累积大量幽灵连接？

这是 **强制杀进程 + WebSocket 没有优雅关闭** 的组合结果：

1. Electron 用 `taskkill /F /T /PID` 强制杀掉 Python 进程
2. `/F` 让进程瞬间死亡，Python **来不及发送 WebSocket 关闭帧和 TCP FIN**
3. 结果：Electron 客户端的连接卡在 `CLOSE_WAIT`，服务端连接卡在 `FIN_WAIT_2`
4. 这些连接最终沉入 `TIME_WAIT`（PID 变成 0），导致你每次重启都「制造」一批新尸体

## 各状态的真正含义

| 状态 | PID | 本质 | 是否需要处理 |
|---|---|---|---|
| `LISTENING` | `19652` | Python 后端在监听端口 | **唯一需要杀的** |
| `FIN_WAIT_2` | `19652` | Python 被强杀后的残留半开连接 | 进程已死，系统自会回收 |
| `CLOSE_WAIT` | `11164` | Electron 客户端没来得及关闭的连接 | 进程退出后消失 |
| `TIME_WAIT` | `0` | 正常断连后的内核残留 | **无需处理，自动消失** |

## 已修复的问题根因

[`backendManager.js`](frontend/electron/managers/backendManager.js:64) 的 `getPidByPort` 之前不加过滤，把 `CLOSE_WAIT` 客户端、`TIME_WAIT` 幽灵连接（PID 0）全当成「占用者」去杀，导致对着 PID 0 和 Electron 自身死循环到超时。

已修改：
- `getPidByPort`：Windows 下只收集 `LISTENING` 状态，并过滤 `PID === 0`
- `waitForPortRelease`：额外过滤 `PID === 0`，避免误杀和超时死循环

如果你想进一步减少幽灵连接，可以把 Electron 退出流程改成：先给 Python 普通终止信号让它优雅关闭 2~3 秒，超时未退出再补 `taskkill /F`。

## 问题原因

后端进程残留主要有 **3 个原因**：

1. **Windows 下 `taskkill` 没有杀子进程树**
   原代码执行 `taskkill /F /PID ${pid}`，只会杀掉指定的 Python 主进程，不会递归杀掉它的子进程（如 uvicorn 产生的工作进程）。子进程在父进程被杀后变成孤儿进程，继续占用 `8000` 端口。

2. **`stop()` 是 callback 异步风格，调用方没有等待**
   `main.js` 在 `window-all-closed` 和 `before-quit` 事件里调用 `backendManager.stop()` 后，立刻执行 `app.quit()`，Electron 进程在 stop 的端口轮询完成前就退出了，导致收尾逻辑被中断。

3. **开发环境启用了 uvicorn `reload=True`**
   `main.py` 在开发环境（`frozen=False`）下启用了 `reload=True`，uvicorn 会额外启动一个文件监控子进程来监听代码变更。这个子进程同样会在主进程被杀后残留，变成额外的孤儿进程。

## 修改内容

### 1. `frontend/electron/managers/backendManager.js`

- **Windows `taskkill` 加 `/T` 参数**：`taskkill /F /T /PID ${pid}`，强制递归杀掉整个进程树，防止子进程残留。
- **`stop()` 重构为 `async/Promise`**：调用方可以用 `await` 等待停止完成，确保在退出 Electron 前已经清理完毕。
- **增加 `waitForPortRelease()` 方法**：轮询检测端口是否被释放，最多等待 5 秒，并兜底杀掉任何仍占用端口的 PID。
- **`restart()` 改为 `async`**：先 `await this.stop()`，确认端口释放后再 `this.start()`，避免竞态导致新旧进程同时存在。

### 2. `frontend/electron/main.js`

- 将 `window-all-closed` 和 `before-quit` 事件处理函数改为 `async`：
  ```js
  app.on('window-all-closed', async () => {
    await backendManager.stop();
    terminalManager.killAllTerminals();
    app.quit();
  });
  ```
  保证 Electron 在真正退出前，后端服务已经停止完成。

### 3. `main.py`

- 移除 `if getattr(sys, 'frozen', False)` 的分支判断，统一使用：
  ```python
  uvicorn.run(app, host=..., port=..., reload=False, ...)
  ```
  开发环境和打包环境都禁用 `reload`，避免 uvicorn 的文件监控子进程造成额外的进程残留。同时统一直接传入 `app` 对象，不再使用 `"main:app"` 字符串，行为更一致。

---

## 效果

重启或关闭 Electron 时，Windows 会递归终止 Python 进程树，且 Electron 会等待确认 `8000` 端口彻底释放后才退出，不会再出现多次重启后堆积多个后端进程的问题。
