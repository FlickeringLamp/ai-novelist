# Electron 打包问题总结

日期: 2025-04-04

## 问题背景

执行 `npm run build-electron` 时遇到的打包问题。

---

## 问题 1: node-gyp 找不到 Visual Studio

### 错误信息
```
Error: Could not find any Visual Studio installation to use
    at VisualStudioFinder.fail
    at VisualStudioFinder.findVisualStudio
```

### 原因
- 项目依赖了 `node-pty`（终端功能）
- `node-pty` 是原生 C++ 模块，需要编译
- `electron-builder` 使用 `@electron/rebuild` 自动重建原生模块
- `@electron/rebuild` 内部的 `node-gyp` 默认只识别 VS 2017/2019/2022
- 用户的 VS 18 (Community) 版本号不被识别

### 尝试的解决方案

#### 方案 1: 环境变量 (失败)
```powershell
$env:GYP_MSVS_VERSION="2022"
$env:npm_config_msvs_version="2022"
```
- 结果: 不生效，@electron/rebuild 仍找不到 VS

#### 方案 2: 配置 electron-builder.json (失败)
添加 `"buildDependenciesFromSource": false, "nodeGypRebuild": false`
- 结果: 不生效，@electron/rebuild 仍然执行

#### 方案 3: 命令行参数 (成功)
```powershell
npx electron-builder --config.npmRebuild=false --win
```
- 使用 `--config.npmRebuild=false` 强制跳过 npm 重建
- 注意: `--config.electronRebuild=false` 是错误的参数名，会导致配置验证失败

---

## 问题 2: winCodeSign 解压权限错误

### 错误信息
```
ERROR: Cannot create symbolic link : 客户端没有所需的特权。
Path: C:\Users\...\electron-builder\Cache\winCodeSign\...\darwin\10.12\lib\libcrypto.dylib
```

### 原因
- `electron-builder` 需要下载 `winCodeSign` 工具进行代码签名
- 该工具包内包含符号链接（macOS 的 .dylib 文件）
- Windows 创建符号链接需要管理员权限

### 解决方案
**以管理员身份运行 PowerShell/VS Code**

1. 关闭当前终端/VS Code
2. 右键点击 → "以管理员身份运行"
3. 重新执行打包命令

---

## 最终可用命令

```powershell
cd frontend
npm run build
npx electron-builder --config.npmRebuild=false --win
```

或使用 package.json 中的脚本：
```json
{
  "scripts": {
    "build-electron": "npm run build && electron-builder --config.npmRebuild=false"
  }
}
```

---

## 注意事项

1. `--config.npmRebuild=false` 会跳过所有原生模块的重建
2. 如果 `node-pty` 与 Electron 版本不兼容，终端功能可能无法使用
3. 生产环境建议配置完整的 VS 环境让重建正常执行

我们项目打包后似乎会无限启动，后台挂满12个进程？
经典的 PyInstaller + multiprocessing 问题。当使用 PyInstaller 打包后，如果代码中使用了 multiprocessing（或者是像 ChromaDB、LangChain 这类库内部使用了 multiprocessing），而没有正确处理 multiprocessing.freeze_support()，就会导致子进程反复启动主程序，形成无限循环。

解决方案很简单：在 main.py 的最开头（任何其他导入之前）添加 multiprocessing 的 freeze_support。
原因： PyInstaller 打包后，multiprocessing 的子进程会重新执行整个 .exe，导致无限递归启动。

依赖为什么不能直接用.venv
而是每次重新下载呢？

因为 **`.venv` 和嵌入式 Python 完全不兼容**。

主要原因：

1. **Python 版本不同**
   - `.venv` 基于你开发环境的 Python（3.13.9）
   - 便携包使用**嵌入式 Python**（官网下载的独立版本）
   - 两个 Python 的安装路径、DLL 位置完全不同

2. **路径硬编码**
   - `.venv` 里的包有**绝对路径**，如：
     ```
     #!F:\盘备份\ai-novelist\.venv\Scripts\python.exe
     ```
   - 复制到便携环境后，这些路径失效

3. **嵌入式 Python 的特殊机制**
   - 使用 `python311._pth` 文件控制模块搜索路径
   - 不使用标准的 site-packages 结构
   - 需要用 `--target` 参数**重新安装**到正确的位置

4. **跨机器兼容性**
   - `.venv` 包含与开发机相关的二进制（可能链接到系统 DLL）
   - 便携包需要在任何 Windows 机器上运行，必须干净独立

**这就是为什么脚本要用 `--target` 重新安装：**
```powershell
pip install --target="$BuildDir\python\Lib\site-packages" -r requirements.txt
```

这样可以确保所有依赖都在正确的位置，与嵌入式 Python 兼容。
