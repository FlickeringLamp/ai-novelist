# 问题：
这个项目中，AI调用工具，创建文件时，
如果是流式信息
假设文件名为 test.txt

ai传来的是
t（第一次）
te（第二次）
text（第三次）
text.txt（第四次）

这个时候它是如何处理的？
怎么过滤掉前面这些没用的chunk


# 分析

根据代码分析，这个项目(Roo Code)处理流式工具调用中不完整文件名的核心机制如下：

问题场景
当 AI 以流式方式调用 write_to_file 工具时，参数通过多个 chunk 传输：

chunk 1: {"path": "t"}
chunk 2: {"path": "te"}
chunk 3: {"path": "text"}
chunk 4: {"path": "text.txt"}

如果直接使用这些不完整的值，会导致文件路径错误。

解决方案：路径稳定检测
项目在 BaseTool.ts 中实现了 hasPathStabilized 方法：

protected hasPathStabilized(path: string | undefined): boolean {
    const pathHasStabilized = this.lastSeenPartialPath !== undefined && this.lastSeenPartialPath === path
    this.lastSeenPartialPath = path
    return pathHasStabilized && !!path
}

工作原理
调用次数	传入的 path	lastSeenPartialPath	返回值	说明
1	"t"	undefined	false	首次收到，不稳定
2	"te"	"t"	false	与上次不同，还在变化
3	"text"	"te"	false	与上次不同，还在变化
4	"text.txt"	"text"	false	与上次不同，首次完整
5	"text.txt"	"text.txt"	true	连续两次相同，已稳定！
在 WriteToFileTool 中的应用
在 WriteToFileTool.ts 的 handlePartial 方法中：

override async handlePartial(task: Task, block: ToolUse<"write_to_file">): Promise<void> {
    const relPath: string | undefined = block.params.path
    let newContent: string | undefined = block.params.content

    // 等待路径稳定后才显示 UI（防止显示截断的路径）
    if (!this.hasPathStabilized(relPath) || newContent === undefined) {
        return  // 路径还在变化，直接返回，不处理
    }
    
    // 只有路径稳定后才继续执行...
}

关键设计点
连续相同值检测：不要求解析 JSON 结构或验证扩展名，只检测"连续两次收到的值是否相同"

为什么有效：

流式 JSON 解析器（partial-json）在 chunk 边界切割字符串时会返回截断值
当模型完成写入一个完整值后，后续 chunk 会继续传递该完整值
连续两次相同意味着模型已写完该参数
状态重置：在工具执行完成后调用 resetPartialState() 清理状态，为下次调用做准备


任务完成

这个项目通过**路径稳定检测（Path Stabilization）**机制来过滤流式传输中的不完整 chunk。

核心机制
在 src/core/tools/BaseTool.ts 中实现了 hasPathStabilized 方法：

protected hasPathStabilized(path: string | undefined): boolean {
    const pathHasStabilized = this.lastSeenPartialPath !== undefined && this.lastSeenPartialPath === path
    this.lastSeenPartialPath = path
    return pathHasStabilized && !!path
}

过滤逻辑
对于你的例子（test.txt）：

第1次收到 "t" → 与上次(undefined)不同 → 不处理
第2次收到 "te" → 与上次 "t" 不同 → 不处理
第3次收到 "text" → 与上次 "te" 不同 → 不处理
第4次收到 "text.txt" → 与上次 "text" 不同 → 不处理
第5次收到 "text.txt" → 与上次相同 → 处理！

实际代码：


```ts
export const createPathStabilizer = (): ((path: string | undefined) => boolean) => {
  let lastSeenPath: string | undefined = undefined; // 闭包变量

  return (path: string | undefined): boolean => { // 返回的函数
    const pathHasStabilized = lastSeenPath !== undefined && lastSeenPath === path;
    lastSeenPath = path; // 更新闭包变量
    return pathHasStabilized && !!path;
  };
};

```
关键点：lastSeenPath 定义在外层函数中，但内层返回的函数可以访问并修改它。即使外层函数执行完毕，这个变量也不会被销毁。


lastSeenPath !== undefined && lastSeenPath === path;
lastSeenPath !== undefined：排除第一次调用（此时还没有历史值）
lastSeenPath === path：当前路径与上一次是否相同


第一次调用时，lastSeenPath 是 undefined（初始值），所以：

lastSeenPath !== undefined → undefined !== undefined → false
由于是 && 运算符，第一个条件为 false，整个表达式就是 false
pathHasStabilized = false
所以第一次调用返回的是 false。

部分	含义
pathHasStabilized	是否连续两次收到相同的路径（已稳定）
!!path	双重否定将 path 转为布尔值：undefined/""/null → false，有值 → true


用法：
用ref引用，确保使用同一个函数，才能使用正确的闭包变量
