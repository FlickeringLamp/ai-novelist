# 启动器与主项目的关系

## 目录结构（分发时）

```
launcher.exe          <-- Go 启动器（Wails 桌面应用）
qingzhu/              <-- 从 git 拉取的主项目（即 ai-novelist）
    launcher/
    backend/
    frontend/
    scripts/
    data/             <-- 运行时生成的数据目录
        config/
            store.yaml
            skills.yaml
        chromadb/
        db/
        uploads/
        temp/
        skills/
        .aiignore
        .gitignore
        .userignore
```

## 关键对应关系

| 概念 | 说明 |
|------|------|
| `ai-novelist` | 本仓库的原始名称，开发时的项目名 |
| `qingzhu` | 分发时的项目目录名，启动器从 git 下载后存放的文件夹 |
| `launcher.exe` | 最终交付给用户的唯一可执行文件 |

- `qingzhu` 是 `ai-novelist` 的发布/分发名称
- 启动器通过 `config.yaml` 中的 `git.project_dir: "qingzhu"` 确定下载目标文件夹名
- 所有路径计算都基于 `exe 所在目录 / qingzhu`

## 启动流程（简化）

```
launcher.exe 启动
    │
    ▼
检查并下载 Python、Node.js（放在 exe 同级缓存目录）
    │
    ▼
检查 qingzhu/ 是否存在
    │
    ▼
【配置迁移】（由启动器负责，不再由后端负责）
    ├─ 从 qingzhu/scripts/store_migration.yaml 读取默认值
    ├─ 补全 qingzhu/data/config/store.yaml 缺失字段
    ├─ 从 qingzhu/scripts/skills_migration.yaml 读取默认值
    ├─ 补全 qingzhu/data/config/skills.yaml 缺失字段
    ├─ 检查 data/ 下 6 个一级目录是否存在，没有则创建
    └─ 检查 .aiignore / .gitignore / .userignore 是否存在，没有则从 dotfiles_migration.yaml 创建
    │
    ▼
部署后端（创建 venv、pip install）
    │
    ▼
部署前端（npm install）
    │
    ▼
启动 Electron（Electron 内部会自行启动 Python 后端）
```

## 为什么迁移逻辑放在启动器而不是后端

1. **单一职责**：启动器是用户唯一接触的入口，环境准备应由它负责
2. **避免重复**：后端初始化时不再做配置迁移，避免启动器和后端做同样的事
3. **时序正确**：后端启动前，配置必须已经就绪（store.yaml 中的端口、模型配置等）

## 相关代码位置

- 启动器迁移逻辑：`launcher/internal/migration/migration.go`
- 启动流程集成：`launcher/internal/launcher/launcher.go`（第 4 步）
- 后端已移除迁移：`backend/settings/initializer.py`（仅剩目录创建和 git 初始化）
